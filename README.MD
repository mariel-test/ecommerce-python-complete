# Ecommerce Automation вҖ“ UI + API

Este proyecto combina pruebas de API y pruebas de UI para un sitio de ecommerce.

## Setup

```bash
cd /home/mariel/MyProject/ecommerce
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# Editar .env con tus datos
```

## Ejecutar los tests

```bash
# Todos los tests
pytest tests/ -v

# Solo pruebas de API
pytest tests/api/ -v

# Solo pruebas de UI
pytest tests/ui/ -v

# Generar reporte HTML de pytest
pytest --html=python-report/report.html --self-contained-html

# Ejecutar tests y abrir el reporte automГЎticamente
python3 run_tests_open_report.py
```

- `tests/api/` para pruebas de servicios REST.

## Cobertura de casos y decisiones estratГ©gicas

[Ver Reporte Interactivo](https://mariel-test.github.io/ecommerce-paython-complet/ecommerce_covarege_test_case.html)

- El set completo suma 52 casos de prueba: 15 de Usuarios, 10 de AutenticaciГіn, 10 de Productos, 5 de Marcas, 5 flujos E2E y 12 de UI/IntegraciГіn.
- La lГіgica de triaje por tiempo funciona asГӯ:
  - Con tiempo al 50% (30 TC) ejecutГЎs todos los S1 mГЎs los S2 del camino crГӯtico y los 5 flujos E2E. Es el punto de equilibrio ideal: cubrГӯs el 100% de lo que puede bloquear un release y el ~85% del riesgo de negocio real.
  - Con tiempo al 25% (16 TC) bajГЎs a solo el camino feliz de cada mГіdulo mГЎs E2E-01 y E2E-04. Esto confirma que el flujo de conversiГіn principal estГЎ vivo, pero deja expuesto el manejo de errores.
  - El smoke de 8 TC es la red de seguridad mГӯnima antes de un deploy: 1 caso por mГіdulo + el E2E completo de ciclo de vida. Si algo falla acГЎ, el release no sale.
- Regla de oro para el triaje: los S1 no se negocian. Los que se pueden diferir sin demasiado riesgo son los casos de mГ©todo no soportado (405), los de borde con inputs muy largos o caracteres raros, y los de consistencia cross-endpoint, porque esos suelen ser mГЎs estables una vez que el camino crГӯtico pasa.
